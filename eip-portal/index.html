04 三個核心設計決定

這三個決定讓平台低風險、快見效、對老系統零傷害。

1
模組只讀「整合資料庫」,不直連來源系統。

中間隔一層乾淨副本,ETL 排程把各來源同步進來。來源掛了 Portal 照跑、老系統零改動、零風險——這是整個架構最重要的一條界線。

2
第一版只做「查詢與看板」,不做回寫。

審批、下單這類寫回來源的功能留給 v2。先做讀,價值最快、風險最低,也不需要碰來源系統的交易邏輯。

3
逐模組上線,不 big-bang。

先把地基(SSO+RBAC)打好,再從決策者最有感的財務看板切入,讓案子先活下去,其餘模組再滾動推進。

STEP 203 財務看板決策者最有感

05 MVP:最小可行方案

用「一條資料管線 + 一個看板 + SSO/RBAC 地基」證明整個架構的核心假設,再談其他模組。

✓ MVP 做這些

  • 地基:Laravel 12 + SSO(先接一個 IdP)+ RBAC(2–3 個基本角色)
  • 一條資料管線:挑最容易唯讀直連的一個來源系統 → 整合資料庫 → 排程 ETL(每日增量)
  • 一個看板:財務決策看板 或 營運即時查詢 擇一,做決策者最有感的那頁
  • 部署:CI/CD + dev / prod + 整合庫每日備份

✗ MVP 先不做

  • 其餘四個模組(先證明一個)
  • 任何回寫來源系統(只讀)
  • 資料列級的複雜權限
  • 多來源整合(先一個來源)
  • 圖面 / 文件處理

驗收標準(做到就算 MVP 成功):

✓ 決策者用公司帳號 SSO 登入 · 看到一個從老系統整合來的即時看板 · 資料每天自動更新且有筆數對帳 · 不同角色看到不同資料範圍(RBAC 生效)。

為什麼這樣切:一次驗證三個最大風險——SSO 能不能接、來源能不能唯讀整合、看板對決策者有沒有價值。這三關過了,其餘模組就是照同一套路複製擴充。

MVP1 來源 + 1 看板證明核心假設
→
v1補齊看板模組多來源 + 財務/營運全看板
→
v2回寫 + 進階審批/下單寫回、細權限

06 待釐清

起案時列的問題,跟合作夥伴對齊後填空。

  • 這是公司案還是 side project?甲方是誰、決策者是誰?
  • 現況:既有 EIP 是什麼(自建 / 套裝)?最痛的三個問題?
  • 資料來源逐一盤點:版本、owner、有無 API、能否唯讀直連(要實測 SELECT 1)
  • SSO 現有 IdP 是什麼(AD / Entra ID)?
  • 團隊與時程:幾個人、多久、預算?
  • 為什麼選 Laravel?(團隊既有技能?)
