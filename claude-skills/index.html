Claude Code 實戰好用的
Skills 與 Agents 推薦

這不是「Claude Code 有哪些外掛」的窮舉清單。這裡只放真的在正式使用中發揮作用的東西——每一項都配一個真實發生過的場景,不是功能說明書。讀者預設已經在用 Claude Code,所以直接講重點,不解釋基礎概念。

TL;DR 先講結論

  • superpowers 逼自己「沒有證據不准說完成」,大幅減少 AI 自稱做完但其實沒做完的問題。
  • security-audit 有一關專門「反過來想辦法推翻自己的發現」,只回報真的能被利用的洞——實測抓到一個沒做限流的 API endpoint。
  • Cloudflare 全家桶 寧可查即時官方文件也不信自己記憶——結果查到 Analytics Engine 的 SQL 不支援 quantile() 和 CASE WHEN,不然監控會悄悄壞掉都不會報錯。
  • Code Review 流程抓到一個 SQL 語法錯誤,而且是單元測試全綠也抓不到的那種——不修的話正式站的告警會永遠是啞的。
  • 派工模式把大量讀檔、掃 repo 丟給 subagent,主線只收結論,還能平行跑,省下大量 context。
01

Skills(裝了就能用)

Skill 是給 Claude Code 的「操作手冊」,裝上之後模型會在對的時機自動載入、照裡面的方法做事。以下六個都是真的用出成果的,不是隨手裝著好看的。

superpowers

紀律工具箱

一整套開發紀律的 skill 包:TDD、系統化除錯(systematic debugging)、還有最關鍵的「完成前必須驗證」(verification-before-completion)。核心精神就一句話:沒有證據不准說完成——要求先跑出測試輸出、read-back 或實跑結果,才能宣稱工作做完。

實戰場景

裝了之後,AI 自己說「做完了」卻其實沒做完的狀況明顯變少——因為它被要求先貼出證據,而不是直接下結論。這條規則現在也直接寫進日常工作習慣裡:完成宣告必須附本次執行產生的證據。

github.com/obra/superpowers ↗

security-audit

六階段稽核

多 agent 分工的資安稽核流程:偵察(recon)→ 找洞(hunt)→ 對抗性驗證(adversarial verification,專門設計來反過來證偽自己剛找到的漏洞,殺掉 false positive)→ 出報告。只回報「真的可以被利用」的問題,不是理論上有風險就列進去。

實戰場景

拿去掃自架的個人 SaaS 作品集網站和幾個 side project,抓到沒做限流的 API endpoint,後來就修掉了。因為有對抗性驗證那一關,報告裡不會塞一堆嚇人但其實不影響的雜訊。

github.com/cloudflare/security-audit-skill ↗

Cloudflare 全家桶

部署主力

官方 skill 系列(cloudflare、wrangler、workers-best-practices 等),是部署個人專案的主力工具。它的特性是寧可即時查官方文件,也不信模型訓練時記住的舊知識。

實戰場景

幫忙接 Cloudflare Analytics Engine 監控時,因為它去查了即時文件,才發現該產品的 SQL 方言不支援 quantile() 和 CASE WHEN。這種坑不會報錯,只是監控部署後悄悄沒有資料——沒查即時文件的話很可能就直接沿用舊知識踩下去了。

github.com/cloudflare/skills ↗

Webwright

瀏覽器自動化

讓 Claude 用 Playwright 開一個真的瀏覽器去做網頁任務:查機票、爬資料、填表單,每一步都留截圖和操作紀錄(action log)方便事後核對。

實戰場景

拿來做一個每天定時跑的 Google Flights 機票比價任務——排程自動開瀏覽器查、比對價格變化,不用自己每天手動盯著看。

github.com/microsoft/Webwright ↗

design-taste-frontend

反罐頭設計

一個「反樣板感」的前端設計 skill,做 landing page、作品集、教學頁這類需要視覺判斷的東西時會用它——目的是避免輸出長得像 AI 預設模板。你現在正在看的這個頁面,就是它的產出之一。

本機安裝的 skill,目前沒有可靠的公開 repo 連結,故不附連結。

context7

即時文件

一個查「版本正確」官方文件的 MCP/skill,專治「AI 用了過時 API」這個老毛病——寫程式前先查現在這個版本的 API 長什麼樣,而不是憑訓練時記住的舊版介面亂寫。

上游開源專案:github.com/upstash/context7 ↗

02

Agents 與用法心法

這節談的不是「裝哪個外掛」,是怎麼用 Claude Code 內建的 agent 機制,把重複的工作模式變成可以一直重用的流程。

Code Review 必過流程

每一次程式碼類的改動,都要先過 code review(內建 /code-review 指令或一個 Code Reviewer agent)才算完成,不是自己看一看順眼就送出去。

! 單元測試全綠,不代表沒問題

有一次 code review 抓到一個監控系統裡的 SQL 語法錯誤。如果沒被抓到,部署上去的查詢會永遠失敗,告警會悄悄地一直是啞的——但整個過程單元測試從頭到尾都是綠燈,因為測試根本沒有真的打過那條 SQL。這是「跑測試」跟「code review」抓的是不同種類的錯的活生生案例。

派工模式:Explore / general-purpose subagent

掃 repo、大量讀檔、做研究這種吃 context 又耗時間的工作,丟給 subagent 去做,主線對話只收「結論」。好處有兩個:主線的 context 不會被一堆中間過程塞爆,而且多個 subagent 可以平行跑,不用排隊等。

把重複需求固化成自訂 agent

常見的例子:一個 Telegram 訊息分類/派工的「管家」agent,或某個領域的專家 agent。把 system prompt 寫好一次,之後每次只要一句話呼叫,不用每次重新解釋一遍背景脈絡——實測下來比每次重講一次前因後果快上一個數量級。

03

加碼工具

不算 Claude Code 的 skill/agent 生態,但幾乎每次要把文件餵給 AI 之前都會先用它。

markitdown

文件轉檔 CLI

把 PDF、Word、Excel、PPT 轉成 Markdown 的命令列工具。任何要餵文件給 AI 讀的場合,第一步幾乎都是先跑這個,轉乾淨再丟進對話。

04

安裝小抄

Claude Code 的 plugin 指令確實可用的是這兩條:claude plugin marketplace add <來源>(從 GitHub repo / URL / 路徑加一個 marketplace)和 claude plugin install <plugin>@<marketplace>(從已加入的 marketplace 安裝)。至於每個專案確切的 plugin/marketplace 名稱,以下沒把握的一律寫「詳見官方 README」,不亂猜參數。

superpowersobra/superpowers
Plugin。claude plugin marketplace add obra/superpowers 之後 claude plugin install,確切 plugin 名稱詳見官方 README。也可能已包含在 Claude 官方 plugin 目錄裡。
security-auditcloudflare/security-audit-skill
Skill 檔案。複製到 ~/.claude/skills/,或詳見官方 README 的 plugin 安裝方式。
Cloudflare 全家桶cloudflare/skills
Skill 檔案。把需要的資料夾(cloudflare、wrangler、workers-best-practices …)複製到 ~/.claude/skills/。
Webwrightmicrosoft/Webwright
Plugin。claude plugin marketplace add microsoft/Webwright 接著 claude plugin install webwright@webwright。
design-taste-frontend
Skill 檔案,直接放進 ~/.claude/skills/design-taste-frontend/。沒有可靠公開 repo,故不附連結。
context7upstash/context7
通常已包含在 Claude 官方 plugin 目錄裡可直接 claude plugin install context7;若要接原始 MCP server,詳見官方 README。
Code Review 流程
通常已內建於官方 plugin 目錄,直接用 /code-review 指令,不需額外安裝。
markitdownmicrosoft/markitdown
