今天的一句話

Agent 元年進入「治理與生產化」階段——工具大家都有了,分出高下的是隔離、稽核、成本控制與組織設計。

三個大趨勢

  1. Frontier Agents 產品化:DevOps Agent(調查 GA/修復還沒)、Security Agent(pen testing/code review GA,threat modeling/code vulnerabilities preview)、Kiro(spec-driven+Powers)。AWS 把「自主 agent」當產品線在賣。
  2. 架構收斂:三個台灣案例(趨勢科技、幣託、NextLink)同一配方——EKS 多租戶(隔離)+ Bedrock(模型不出帳號)+ MCP(工具接口)+ Skills(知識注入);上下文三層(Instructions/Skills/Memories)與 Claude Code 同構。
  3. 人機分工定型:「AI 跑腿彙整、人做決策」全天重複三次(OpenClaw×DevOps Agent、Wingman TAM、ASKUL 併進型)。

最有料的五個技術點

  1. K8s RuntimeClass 沙箱:runc/gVisor/Kata 同 cluster 混用,一行 yaml 換隔離等級;2026/2 起 C8i/M8i/R8i 巢狀虛擬化讓 Kata 免 bare-metal。
  2. agent-sandbox-controller(CRD 仿 PV/PVC):Template/Claim/Sandbox/WarmPool,預熱池解 idle 成本。
  3. AI Gateway 模式:模型前自建一層集中管 token 驗證/健康/quota/路由/重放(INF02);幣託版=LiteLLM 零信任路由+Langfuse 觀測+Pod Identity 零金鑰。
  4. ALB Target Optimizer:push→pull 的併發感知 LB,GPU/LLM 後端尾延遲治理(100% in-SLA 實測);WebSocket 不支援。
  5. Quota 決策階梯:60% 申請調升/80% 換路由/90% 優雅降級——可直接抄成告警閾值。

KIRO01 — AWS DevOps Agent 自主事件調查(11:30,201ABC)

Operations is a context problem

  • 73% 的事件處理時間花在蒐集上下文,而非修復問題。
  • 單一 Sev2 事件,值班工程師要用 11+ 個工具。
  • 3am 是人類最不擅長關聯訊號的時刻。
  • 分工主張:DevOps Agent 負責蒐集,工程師負責決策。
  • 可借鏡:73% 這個數字適合包裝 devbox watchdog(自動蒐集 error rate/p95)的動機。

事件生命週期五階段(標準 SRE 鋪陳)

  1. 偵測(判斷嚴重度、是否叫醒團隊)
  2. 調查(蒐集上下文、驗證假設以指導恢復)
  3. 恢復(快速恢復,平衡速度 vs 產生額外問題的風險)
  4. 事後檢討(無責 RCA、跨系統改善機會)
  5. 改善(強化可觀測性、測試、基礎設施)

DevOps Agent 四大核心能力

  1. 訊號接收:CloudWatch/X-Ray/EventBridge/Datadog/Dynatrace/GitHub/PagerDuty → 統一單一事件時間軸。
  2. 拓撲:即時服務與資源相依圖,agent 持續維護(什麼呼叫什麼、何時變更)。
  3. 根因分析:從症狀反向走訪拓撲;引用證據排序候選原因;排除干擾項。
  4. 詳細緩解計畫:每選項附理由、可逆性、風險;禁止措施先過濾(如營業時間不做容錯切換)。
  • 觀察:第 4 點的「可逆性+營運護欄」是敢讓 agent 碰 production 的關鍵設計。

Agent 運作架構(全圖)

  • 輸入訊號:AWS 原生(CloudWatch/X-Ray、EventBridge/CloudTrail、CW Logs/CodePipeline)+ 第三方 APM(Datadog/Dynatrace、New Relic/Splunk、Grafana/Prometheus)+ 開發與值班(GitHub/GitLab、PagerDuty/Opsgenie、Jira/自訂 webhooks)。
  • 推理核心八模組:訊號接收(正規化/關聯)、推理迴圈(Bedrock 模型)、程式碼索引(儲存庫/分支/功能)、Runbook 檢索(向量+結構化)、技能目錄(宣告式規格)、緩解規劃器(計畫+理由)、記憶與狀態(每事件上下文)、MCP 伺服器(工具對外提供)。
  • 介面:Slack 機器人(告警+對話問答)、Kiro IDE(MCP client)、Quick console(MCP client)。
  • 產出:含理由的緩解計畫、Runbook+差異建議、值班呼叫、稽核日誌+事件筆記。
  • 觀察:①對外整合全押 MCP(Anthropic 的協議成 AWS 官方標準件)②技能目錄=宣告式規格,與 Claude Code skills 同構;整張圖可當 agent 系統設計參考。

RCA 七個調查動作(推理迴圈按需呼叫,各附工具組合)

  1. 指標分析:症狀時間軸/異常偵測/錯誤率延遲量化 — use_aws cloudwatch、pi、sqs、datadog_get_metric
  2. 日誌搜尋:錯誤訊息/堆疊/慢查詢/權限失敗 — query_cloudwatch_logs、describe_log_groups
  3. 基礎設施變更調查:IAM/Lambda/CFN 變更×事件時間關聯 — lookup_cloudtrail_events
  4. 程式碼探索:拒絕策略/查詢生成/斷裂參照 — repo_grep、repo_read、git_log、git_diff
  5. 部署與程式碼審查:CFN 事件、CI/CD、GitHub 提交記錄看最近部署
  6. AWS 資源檢查:IAM 角色、Lambda 設定、SQS 屬性、事件來源對應
  7. 調查編排:規劃任務、綜合發現、跨 Agent 判定根因(→ 底層是多 agent 協作)
  • 本質:把資深 SRE 的 debug 直覺拆成七個可呼叫動作。

系統概覽

  • 邊界:目前僅 US-EAST-1(台灣落地注意延遲/資料出境);認證 AWS IDC/IdP;操作員 Web 應用(拓撲/調查/對話/預防/技能)+管理控制台;後端 Bedrock 驅動。
  • Agent 類型六種:通用、隨需、事件分級、事件 RCA、事件緩解、評估 → 印證多 agent 分工(分級/調查/緩解各自獨立,另有評估 agent 打分)。
  • 整合四路:工單溝通(ServiceNow/PagerDuty/Slack)、自訂 MCP 服務(OAuth 2.0+API Keys,接內部工具的入口)、CI/CD(GitHub/GitLab/Azure DevOps)、可觀測性(Dynatrace/Datadog/Splunk/New Relic)。
  • 調查觸發器:通用 webhook、手動(Web 應用)、ServiceNow Business Rule→webhook、PagerDuty 工單→webhook、Dynatrace Problem→Workflow→webhook(全收斂到 webhook)。
  • 分析與輸出:不可變 Agent 日誌(所有推理步驟,稽核/回放)、歷史事件分析+建議、時間軸+根因分析、緩解計畫、AWS Support Case 自動附上下文。
  • 觀察:不可變推理日誌是企業敢用的前提(事後可回放判斷過程)。

工作空間即時協作(UI 實截圖)

  1. 告警 → Agent 自動發布調查結果(含拓撲圖)。
  2. 討論串中工程師 @-mention 機器人追問。
  3. 緩解計畫附完整理由(demo:停用 chaos engineering 場景)。
  4. 證據連結:日誌/指標/Runbook 內嵌呈現。
  • 體驗定位:「多一個 SRE 同事在討論串裡」;AWS 用 chaos 注入故障來測/展示 agent。

主動事件預防(Ops Backlog)

  • 方法論:每週回顧 7-30 天事件 → 模式識別(時間/資源/部署)→ 產生目標化可執行建議,進 Ops Backlog(每週跑、狀態追蹤、需 review)。
  • 建議四類:①可觀測性改善(補告警/追蹤/日誌彙總)②基礎設施最佳化(資源大小/自動擴展/冗餘)③部署管線強化(安全掃描/金絲雀/整合測試)④應用韌性(斷路器/重試/逾時)。
  • UI 範例:K8s admission controller 擋危險 env var、CloudWatch 告警多期評估消誤報、CI/CD 加 pre-deployment 安全掃描。
  • 觀察:等於把 postmortem action items 自動產生+追蹤,解決「寫完就忘」。

隨需 SRE 任務處理(對話式,平時可問)

  • 對話能力:複雜查詢 NLU、多區域分析、基於對話歷史追問。
  • 深度檢視:架構(拓撲/資源清單/偏移偵測)、部署(CI/CD 健康度/儲存庫/發布速度)、可觀測性(指標關聯/日誌模式/追蹤)、韌性(錯誤趨勢/相依性健康/安全態勢)。
  • 範例問法:容量「接近限制的 Lambda」、安全「30 天內到期憑證」、效能「昨天延遲飆升原因」、成本「CPU<10% 的 EC2」。
  • 對話 demo 的回答結構:結論 → 列出查了什麼(證據)→ 主動提後續選項(好 agent 樣板)。

分級代理(事件分級 Agent)

  • 問題:五告警同時觸發 → 五個獨立調查 → 同一根因,重工。
  • 解法:相關事件關聯成單一調查;已連結事件標「LINKED」不自動另起調查、與主調查分組 → alert storm 去重+火力集中。
  • 對比:Datadog/PagerDuty correlation 類似,差別是去重後直接接自家 RCA agent。

Enterprise-Ready 功能

  • 區域擴展:上線時 6 區(US East 維州/US West 奧勒岡/Frankfurt/Ireland/Sydney/Tokyo)→ 前面「僅 US-EAST-1」是現況,GA 擴 6 區後台灣可用東京。
  • 私有 MCP:連私有 MCP 伺服器、安全存取內部工具、機密流量不經網際網路。
  • 安全性:客戶自管金鑰(CMK)、直接 IdP 整合、Okta、Microsoft Entra ID。
  • 在地化:瀏覽器語言偵測、agent 回應翻譯、偏好語言互動。
  • 觀察:私有 MCP+CMK = 瞄準金融/受監管產業。

Humans stay in control(治理四原則)

  1. 預設唯讀:agent 讀訊號與程式碼;人決定執行什麼。
  2. 計畫、非盲目行動:建議附理由/證據/替代方案;禁止措施先過濾(營業時間不容錯切換、停機期不再平衡)。
  3. 完整稽核+客戶金鑰:推理步驟/訊號/程式碼參照記錄於 CloudTrail;Bedrock+KMS CMK;資料不離客戶區域。
  4. 調查隔離:並行事件獨立;不同標題新告警開全新調查,永不靜默合併。
  • 一句話框架:「AI agent 上 production 怎麼管」= 唯讀預設 → 計畫先審 → 全程稽核 → 隔離爆炸半徑。

韌性=營收(開場數據)

Shared Responsibility Model for Resilience(Well-Architected 官方圖)

  • AWS 管雲「本身」:硬體服務(compute/storage/DB/networking)+全球基礎設施(region/AZ/edge)。
  • 客戶管雲「內」:關鍵基礎設施持續測試、工作負載架構、變更管理與操作韌性、可觀測性與故障管理、網路/限額/約束。
  • 一句話:買 AWS ≠ 買到韌性,上五層自己負責。

韌性 mental model

  • HA:主環境設計抵禦常見故障;指標 SLA。
  • DR:HA 無法處理的故障,災備環境於可承受時間內恢復;指標 RTO/RPO。
  • 持續改善:CI/CD+可觀測性,從部署前測試 → chaos engineering 演練。
  • 三 AZ HA 參考架構:ELB+ASG 跨 AZ、RDS primary/standby/read replica、S3/DynamoDB 原生跨 AZ;服務分級 zonal(EC2/EBS)/regional(ELB/RDS)/global(S3/DDB)——按故障域決定哪層自己做冗餘。

★ AI 工作負載的新故障型態(本場核心)

  • 傳統故障(當機/DB/網路)基礎未變,多 AZ/副本/重試仍必要。
  • AI 新增五種:①模型 throttling/限額耗盡 ②單一模型或單區域強相依 ③Agent 長流程狀態遺失 ④幻覺/不當輸出 ⑤成本失控(token 爆量)。
  • 結論:韌性要做進推論層與 AI Agent 層(額外冗餘+主動彌補);幻覺被歸類為「故障型態」= 輸出品質要像 error rate 一樣監控。

解法 1 — Bedrock CRIS(Cross-Region Inference Service)

  • 打「單區域強相依」:應用打 CRIS 端點,智慧路由自動選健康 region 的 FM、挑資源足/延遲低路徑;支援多 FM+Guardrails。
  • 定位:模型層的 multi-AZ / 官方版模型 fallback。

解法 2 — Quota 管理(「限流即事故」)

  • Token 用量分鐘級追蹤(rolling aggregation),即早偵測限額壓力。
  • 決策階梯:>60% quota 申請調升;>80% 換模型路由/區域;>90% fail gracefully+通知 Ops(可直接抄成告警閾值)。
  • 各模型 burndown rate 不同,估限額要納入。

解法 3 — Context window 約束

  • 應用層先 count_tokens() 守門(放行/截斷),不等模型端爆錯。
  • 參考值:Claude 200K(Haiku/Sonnet/Opus 4.5)、1M(Sonnet 4.6/5、Opus 4.6-4.8、Fable 5);Nova 300K(Lite/Pro)、1M(Lite 2)。

解法 4 — FM Routing Logic(模型層 health check)

  • 四指標:health check、錯誤率門檻、延遲 SLA、限額餘量 → FM Primary/Secondary/Tertiary 三級 fallback。
  • 易漏點:延遲超 SLA 和 quota 見底也算「不健康」,不只看掛沒掛。

  • 功能(2025/11 re:Invent 發表):LB 由 push 反轉為 pull——target 上跑 ALB Agent 追蹤自身併發,「有空位」才向 ALB 要請求;TARGET_CONTROL_MAX_CONCURRENCY(0-1000)。

  • Setup:target 裝 agent(traffic/destination/control port+max concurrency)→ 建 optimized target group(僅能建立時開啟)→ 流量灰度切換(新舊 TG 並存可回退)。
  • 監控:TargetControlWorkQueueLength(ALB 端等待認領的隊列長度,按 AZ)——持續走高=容量不足;適合當 auto scaling 觸發指標。
