SEC02 — AWS Security Agent × CyCraft XecGuard(11:30,101BC)

Why AI security matters(開場數據)

  • 94% 高管:AI 方案部署前要先安全化(IBM 2024)。
  • 96% 技術人員視 AI agents 為成長中風險,但 98% 組織一年內仍計畫擴大使用(Sailpoint 2025)。
  • 97% 通報過 AI 相關安全事故的組織缺乏適當 AI 存取控制(IBM Cost of Data Breach 2025)。
  • 讀法:96%+98%+97% 合起來=「知險仍上+裸奔」。

口號轉向

  • Shift Left Security(Before)→「Security is everywhere」(After 2026):不只左移到開發期,而是全生命週期每層都在。

戰略三段論

  1. AI 將嵌入一切(infra/應用/企業/供應鏈)。
  2. 故安全須嵌入 AI 所到之處(縱深防禦 infra→應用層)。
  3. Securing AI 可反向拉高全企業安全水位(secure-by-design/default AI 逼你盤點存取/資料流/稽核)——AI 安全是槓桿不是負擔。

Real World:AWS 內部 dogfooding

  • Security Agent 抓出內部專案用非標準授權(違反 AWS Authorization/Authentication Standards);工程師:「prototyping 早期改起來 trivial,若三個月後才被 AppSec 抓,extraordinarily painful」——左移價值的最佳量化表述。

★ AI×安全三象限(值得收藏的分類框架)

象限意思對應服務
AI for SecurityAI 幫安全:偵測/自動回應/測試Security Agent、GuardDuty、Detective
Security for AI保護 AI 本身:app/模型/資料/agent 縱深防禦Bedrock Guardrails、AgentCore、Security Hub
Resilience to AI-Powered Threats防 AI 武器化對手Shield Advanced、WAF、Network Firewall、Incident Response
  • 討論「AI 安全」先問是哪個象限,避免混談。

AWS Security Agent GA(併入 AWS Continuum 品牌)

  • 定位:開發生命週期全程主動保護應用的 frontier AI agent;Now GA。
  • 三能力×時間尺度:Design Review(分鐘,設計文件對安全基線)、Code Review(近即時,每 PR 掃)、Pen Testing(小時,多步攻擊場景鏈式驗證)。
  • 不綁 AWS:AWS 任意 region/endpoint、多雲(GCP/Azure/Oracle)、地端 hybrid、SaaS 任何可達端點——搶「安全工具」市場,對 GCP 用戶也可用。
  • Pentesting 拆解(context-aware autonomous):①SAST+DAST+SCA+自主滲測合一 ②Context-awareness(吃設計文件/API/IaC/威脅模型/user stories 客製攻擊)③Validated risks:漏洞鏈成可證明可重現的攻擊(降誤報/告警疲勞)——「鏈漏洞」是人類滲透員核心技能的 agent 化;websec-scanner 的官方對標。
  • Security requirements UI:10 條 managed 基線(稽核日誌/認證/授權/資訊保護/日誌保護/特權存取/密鑰保護/secure-by-default/可信密碼學/租戶隔離)+custom requirements——這份清單可直接當任何系統的安全檢查表。
  • Design Review triage pipeline:意圖對齊→上下文蒐集→風險評估→執行計畫(人在頂/agent 在底);吃完整設計文件。與 DevOps Agent RCA 流程同骨架——AWS 所有 agent 共用同一 pipeline,只換領域知識+工具。
  • Design Review finding demo:設計文件 10 條 9 合規、1 不合規(Secure by Default)——agent 讀懂「為社群互動而預設公開」的商業意圖仍判不安全,建議改私有+明確 opt-in。設計階段(程式碼未寫)即抓到,EIP 設計評審可借力。
  • Code Review demo(進 PR 當 reviewer,GitHub 原生 reply/resolve):抓 encrypt_field 用 Fernet(AES-128)→建議換後量子密碼學(PQC)並抽象化好替換;附 👍👎 評分。亮點:抓的是「加密夠不夠未來」非「有沒有加密」,linter 級工具不會提;零學習成本融入 PR 流程——導入 code review agent 最無痛形態。
  • Pen Testing 三賣點:Autonomous(真實攻擊場景、漏洞可驗證重現)、Application-aware(懂應用脈絡客製測試)、Fast(hours vs weeks+自動產 ready-to-use fix)——壓縮傳統外包顧問+報告流程;與 websec-scanner 差距在「產對的修補」。
  • Pen Testing Phase 1 認證:Login Agent(LLM+headless 瀏覽器)自動登入/填表/處理轉址/存 session→Credential Store(cookies/headers/tokens/AWS profiles);支援帳密/TOTP 2FA(等新視窗)/API key/自訂 header;Multi-actor(admin+user 雙帳號測 IDOR+權限提升)——自動掃描器最難的「懂業務邏輯」越權漏洞。websec-scanner 進階最有價值的一塊。
  • Pen Testing Phase 2 掃描:並行組(Web/API Scanner+TLS/SSL Analyzer+Dependency Scanner CVE)+依賴鏈組(Code Analyzer LLM 靜態→Document Processor 抽端點→Crawler 動態發現);「並行+依賴」編排=與 DevOps Agent RCA/workflow 同思路,整套滲透 agent=一個大 workflow。
  • Pen Testing Phase 3 受管執行:Swarm Worker(LLM agent)並發跑多風險任務;工具箱(代碼執行/注入偵測/瀏覽器自動化/token 操縱/漏洞情報/速率管理);Worker 帶風險類型/憑證/準則/逾時/上下文摘要;輸出 Endpoints(共享狀態)/Findings(驗證)/Resources(清理註冊表)。關鍵:①每次工具呼叫過 4 層 guardrail+違規自我修正(攻擊 agent 自己也被關籠)②cleanup registry 自動清測試殘留——「能力越強、自我約束越關鍵」呼應治理主題。
  • ★ Pen Testing Phase 4 Guided Explorer(最像真人滲透員):Strategist Agent 分析全局(端點/已確認漏洞/程式碼+API spec/Phase 3 歷史/覆蓋缺口)→產架構缺口/鏈接機會/優先目標→swarm 執行→adapt 再重複。adaptive 迭代:首輪(admin 認證繞過/IDOR/檔案上傳 SSRF)→後續看回應演化(拿 admin token 鏈 SQLi/測新 API/error-based 洩漏)。「Tasks evolve based on application responses」=滲透員核心「拿到這個就試那個」的攻擊鏈推理,自動化這層才真取代人類 pentester。
  • Pen Testing Phase 5 清理:Track(Resource Registry 記每個產物:帳號/檔案/資料/設定)→Delete(Cleanup Agent LLM+瀏覽器+bash,5 並行、依類型刪除策略)→Verify(pre/post state 比對確認全清+完整稽核軌跡)。「來過不留痕」把 agent 從炫技 demo 抬升到敢在 production 跑。
  • 五階段全貌 = 資深滲透員完整工作流拆解自動化:認證→掃描→執行→探索(adaptive)→清理。
